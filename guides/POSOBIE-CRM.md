## 1. Зачем нужна CRM

CRM объединяет работу менеджера Legal Consult / Lichka (Казахстан, KZT):

| Задача | Где в CRM |
|--------|-----------|
| План дня: кому звонить, просрочки | **Дашборд** |
| Карточка клиента, анкета, контакты | **Клиенты → карточка** |
| Учёт оплат за документы (отчётность) | **Оплата** в карточке |
| AI: тексты, анализ, заполнение полей | **Jarvis** |
| Юридические PDF | **Редактор PDF** |
| Тренировка отказов регулятора | **Регулятор** |
| Каталог названий документов | **Документы** |

**Клиент не видит CRM и Jarvis.** В переписке клиент общается с **Добряком (Сергей Макаров)**. Jarvis — внутренний помощник менеджера.

## 3. Дашборд

### Что видите

- **Часы Киев / Алматы** — рабочее окно менеджера 10:00–17:00 (Киев); Алматы +3 часа.
- **Command Center** — лента дел по срокам:
  - просрочено / сегодня / завтра / неделя / позже
  - типы: звонок, просрочка оплаты, обещание оплаты, ожидаемый доход, срок документа, задача, «давно не было контакта»
- **Фильтры:** все / звонки / оплаты / задачи / доход
- **Кнопки на карточке пункта:** «Контакт был», «Готово» — скрывает пункт и закрывает связанные задачи Jarvis
- **Jarvis Briefing** — краткая сводка дня (AI, если настроен DeepSeek)
- **Задачи Jarvis** — список открытых задач по всем клиентам
- **Статистика:** клиенты, просрочки, оплаты сегодня (из заказов документов)

### Откуда берутся данные

- Деньги — **только из `client_document_orders`** (не из сделок и не из чата).
- Просрочки — `due_date`, `promised_pay_date`, `payment_status`.
- «Нет контакта» — давность заметок/чата.

## 4. Jarvis — общий чат

**Страница `/jarvis`**

- Чат со **всей базой** и стратегией компании.
- **Не пишет** в карточки клиентов автоматически (только обсуждение).
- Справа — **сводка дня** (briefing).
- Подходит для: «кого в приоритет», «скрипт на возражение», общие вопросы по obych.

Отличие от Jarvis в карточке клиента: там есть **запись в CRM** (поля, иногда заказы по приказу).

## 5. Клиенты

**Страница `/clients`**

### Доска / таблица

- Поиск по имени, телефону, email.
- Вкладки: **Активные** / **Архив**.
- Колонки: имя, телефон, регион, тип клиента, отношение к оплате, долг/оплачено по документам, статус.
- **Создать клиента:** ФИО + телефон.
- **В архив** / **из архива** — данные сохраняются.

### Статусы клиента (поле `status`)

`pending`, `active`, `closed` и др. — этап дела в CRM (не путать с оплатой документов).

### Вкладка «Профиль» (Alt+1)

**Основные поля CRM:**

- ФИО, телефоны (накопительный список), email (накопительный)
- Статус, № дела, этап, прогресс %
- Общие заметки CRM
- **Заметки менеджера** — хронология (не равно оплате)
- **Сделки (deals)** — планы/рассрочки (**не учитываются как факт оплаты**)
- **Анкета** — ~35 кастомных полей (группы ниже)

**Группы полей анкеты:**

| Группа | Поля (примеры) |
|--------|----------------|
| identity | ИИН, паспорт, адрес |
| personal | возраст, регион, семья |
| work | работа, доход офф/не офф, зарплаты |
| credits | кредиты, нагрузка, залог |
| psych | тип клиента, цели, боли |
| **lichka** | потери ₸, к возврату, уже возвращено, всего денег |
| payment | отношение к оплате, возражения, обещанная дата |
| legal | договор/№, история ситуации (брокер) |
| bank | банк, карта, источник средств AML |
| regulator | статус у регулятора |

Jarvis **автоматически** дополняет поля из ваших сообщений в чате (брокер, суммы, контакты, договор).

---

### Вкладка «Оплата» (Alt+2)

**Заказы документов** — единственный источник факта «клиент заплатил нам за услуги Legal Consult».

На каждый заказ:

| Поле | Описание |
|------|----------|
| Название документа | Из каталога или вручную |
| total_amount | Полная сумма |
| client_due | Сколько должен клиент |
| company_pays | Сколько покрывает компания |
| client_paid | **Факт оплаты** |
| payment_status | pending / partial / paid / overdue … |
| work_status | planned / in_progress / done / cancelled |
| due_date, promised_pay_date | Сроки |
| collection_status | waiting / promised / paid / refusing … |
| payment_objection | Возражение клиента |

**Действия:**

- Создать заказ (из каталога или своё название)
- Отметить оплату (`client_paid`)
- Условия оплаты, статус взыскания
- **Редактор PDF** → скачать → отправить клиенту **вручную** (WhatsApp, email)
- Удалить заказ

CRM **не отправляет** PDF клиенту сама.

---

### Вкладка «Jarvis» (Alt+3)

**Внутренний AI-чат по этому клиенту.**

- **Переключатель роли** (видите только вы):
  - Добряк (Сергей) — основной стиль для клиента
  - АРРФР, AFM, Банк, Межд. регулятор, Суд, AML
- Jarvis генерирует **тексты для копирования** клиенту; вы отправляете сами.
- После каждого сообщения — **фоновая запись** в анкету (поля).
- **Заказы на оплату не создаются сами** — только:
  - `Приказ: создай заказ …`
  - `Создай заказ …`
  - `Зафиксируй оплату …`
  - `Внеси в систему оплату …`
  - или вручную во вкладке «Оплата»
- Кнопка **«Анализ и задачи»**: разбор клиента, задачи Jarvis по клиенту
- **«Заказы из истории (приказ)»** — разбор всей переписки (тоже только с императивом для заказов)

**Этап цикла** (лид → консультация → документы → оплата → кейс → закрытие) Jarvis видит в контексте CRM автоматически.

---

### Вкладка «Регулятор» (Alt+4)

- Отдельный чат **симулятор АРРФР** (холодные отказы, KYC, KZT).
- Не смешивается с обычным Jarvis.
- История сохраняется по клиенту.
- Цель: тренировка менеджера и заготовка текстов «от регулятора».

## 8. Регулятор

**Страница `/regulator`**

- Глобальный чат без выбора клиента (можно указать clientId в API).
- Список **типовых требований** регулятора (документы, органы, сложность).
- Стиль: формальные отказы, досудебный порядок, AML, суммы в **тенге**.

**Не путать:**

- Чарджбэк → **банк-эмитент** карты
- Жалоба на брокера → **АРРФР**
- Мошенничество → **правоохранительные органы**, ст. 190 УК РК

## 10. Jarvis

> Подробное пособие: **[POSOBIE-JARVIS-AI.md](./POSOBIE-JARVIS-AI.md)** (20 разделов: роли, цикл, деньги, obych, API, шпаргалка).

### Версия промпта: v5.0 Full Cycle

Jarvis — стратегический ассистент **Сергея Макарова (Добряк)**.

### Что умеет

| Функция | Описание |
|---------|----------|
| Анализ карточки | Заказы, client_paid, анкета Lichka, задачи, заметки |
| Тексты клиенту | 2–3 варианта под выбранную роль |
| Документы | Черновики заявлений, жалоб, договоров → PDF-редактор |
| Сканирование анкеты | Брокер, потери ₸, договор, контакты — из каждого сообщения |
| Следующий шаг | Один конкретный action по этапу цикла |
| База obych | Скрипты, комиссии, возражения, хроники |
| Сводка дня | Briefing на дашборде и `/jarvis` |
| Задачи | Создание/отображение задач по клиентам |
| Регулятор (отдельно) | Симуляция АРРФР |

### Чего Jarvis не делает

- Не создаёт **новых клиентов**
- Не создаёт **заказы/оплату** без императива
- Не **выдумывает** суммы и факты оплат
- Не показывает клиенту **JSON** (только по «покажи JSON»)
- Не **отправляет** сообщения и PDF клиенту

### Роли (внутренние)

| ID | Роль |
|----|------|
| dobryak | Добряк — единственное «лицо» для клиента |
| ardfm | АРРФР РК |
| afm | Финмониторинг |
| bank | Банк (чарджбэк карты) |
| intl_regulator | FCA, MAS, SEC… |
| court | Суд / арбитраж |
| compliance | AML брокера |

API: `GET /api/jarvis/roles`  
В чате: `{ "message": "…", "role": "afm" }`

### Этапы полного цикла

1. Новый лид  
2. Консультация  
3. Сбор документов  
4. Договор  
5. Оплата услуг  
6. Ведение кейса  
7. Закрытие / архив  

Этап вычисляется из CRM и показывается в контексте Jarvis.

### Примеры команд менеджеру

```
Клиент написал: потерял 2 млн на брокере XTrade, тел +7…
→ Jarvis заполнит поля, предложит шаг

Дай текст от банка: клиент не платит 3 дня
→ Выберите роль «Банк», получите 2 варианта

Приказ: создай заказ «Налог 6.5%» на 35000 тенге
→ Создастся заказ в CRM

Покажи JSON
→ Структура полей в ответе (редко нужно)
```

## 11. Два потока денег

| Поток | Где в CRM | Смысл |
|-------|-----------|--------|
| **Lichka — возврат по делу** | Поля `investment_loss_kzt`, `recovery_target_kzt`, `recovery_paid_kzt` | Деньги, которые компания возвращает клиенту по инвестиционному делу |
| **Legal Consult — оплата за документы** | `client_document_orders.client_paid` | Что клиент **заплатил нам** за подготовку документов |

**Jarvis и отчёты смотрят на оплату услуг только по заказам документов.**

Сделки (`deals`), `balance_kzt`, заметки, переписка — **не факт оплаты**.

## 12. PDF и шаблоны

### Процесс

1. Заказ документа в карточке (вкладка Оплата)
2. **Редактор PDF** — выбор шаблона, правка текста
3. Опции: **герб**, **печать**, **подпись**, имя подписанта
4. **Скачать PDF**
5. Отправить клиенту **вручную**

### Шаблоны

- **Универсальный**, справки, договоры — seed в БД
- **dla_posobia** — отчёт регулятора, иск, договор HRL (импорт в каталог)
- Переменные в шаблоне: `{{client_full_name}}`, `{{date_long}}`, `{{client_paid}}`, `{{document_title}}` и др.

### AI в PDF

- Опционально «черновик от AI» при генерации
- Или только подстановка полей без AI

Файлы PDF: `server/storage/generated/` на сервере.

## 15. API

## 18. Модель прав Jarvis (Data Governance)

### Три уровня доверия

| Уровень | Кто | Что может |
|---------|-----|-----------|
| **Jarvis** | AI | Добавлять в пустые поля, дополнять телефоны/email/заметки, создавать **предложения** |
| **Менеджер** | manager/admin | Подтверждать предложения, редактировать некритичные поля |
| **SuperAdmin** | superadmin/supervisor | Прямое редактирование **золотых полей** |

### Золотые поля (Locked)

**Колонки клиента:** ФИО, телефон, email, № дела  

**Анкета:** ИИН, паспорт, потери/возврат ₸, договор, банк, карта, AML, дата обещанной оплаты  

Менеджер **не может** перезаписать их в форме — только SuperAdmin или **Подтвердить** предложение Jarvis.

### Модель «Чат → предложения»

1. Менеджер пишет в Jarvis факты из переписки  
2. Jarvis **не перезаписывает** существующие суммы и золотые поля  
3. При расхождении — карточка **«Jarvis предлагает изменения»** с кнопками Подтвердить / Отклонить  
4. Пустые некритичные поля заполняются сразу (с записью в audit)

### Audit Log

Таблица `audit_log` — каждое изменение: поле, было, стало, кто (Jarvis / Manager), источник. Не удаляется.

API: `GET /api/clients/:id/governance/audit-log`
