# Правила ведения клиента GlobalSafe Finance

Канон для AI и операторов. Синхронизировано с кабинетом financegroup.store, админкой и `docs/RABOTA_AI_KNOWLEDGE_BASE.md`.

---

## A. Жёсткий регламент (STRICT RULES)

1. **Единый источник суммы** — остатки только из полей SDL-кувшинов (`kzt_balance`, `eur_balance`, `usd_balance`, `crypto_balance`, `uk_escrow_frozen`). Не считать «на глаз».
2. **Вывод — полная сумма** — одна активная заявка на весь доступный баланс в выбранной валюте/кувшине; без частичных дублей.
3. **Этапы последовательно** — SDL 1→2→3→4→5→6 без пропусков; форс только через админ с осознанной причиной и логом.
4. **Консолидация перед выводом** — если деньги в нескольких кувшинах, сначала внутреннее объединение (админ `ClientJarService`), затем вывод (в т.ч. **крипто только USDT**, полный остаток на `withdraw_crypto.php`).
5. **Идемпотентность** — повторное действие не создаёт второй активный кейс.
6. **Аудит** — критические правки → `audit_log` / `admin_logs`.
7. **Клиент не видит внутренку** — переводы между кувшинами админом, TX hash, «обход документов», имена таблиц.

## B. Шесть этапов SDL (что видит клиент и что делает оператор)

| # | Клиент в кабинете | progress % | Админ-очередь | Ключевые страницы клиента |
|---|-------------------|------------|---------------|---------------------------|
| 1 | Идентификация клиента | 0–24 | `pipeline_queue.php?stage=1` | `cabinet.php`, `documents.php` |
| 2 | Верификация происхождения средств (SOF) | 25–49 | `sof_review.php` | `source_of_funds.php`, кнопка «Добавить документ» |
| 3 | Юридическая подготовка | 50–74 | stage 3, `identity_review.php` | `documents.php#legal-docs` |
| 4 | Мировые финансовые регуляторы | ~75–77 | `stage_review.php?stage=4` | `cabinet.php`, театр регуляторов |
| 5 | Финансовое урегулирование | ≥78 | `stage_review.php?stage=5` | переход через `advance_sdl_next` |
| 6 | Банковское подтверждение и выпуск | ≥92 | `stage_review.php?stage=6` | финальный выпуск |

**Поля в БД:** `sdl.stage_number`, `current_stage`, `current_task`, `progress_percent`, `funds_status`, `psychotype`, `risk_score`, `global_search_started_at`.

**Опрос кабинета:** `GET /api/identity_pipeline.php` — при смене `stage_number` страница должна перезагрузиться (SSR и JSON не расходятся).

**Переходы:**
- 1→2 — после одобрения identity в `identity_review.php`; уведомление клиенту, кабинет на этап 2.
- 2→3 — SOF пакет согласован в `sof_review.php`, затем `stage_number=3`; фраза клиенту: «Пакет согласован, специалист открывает юридический этап».
- 4→5, 5→6 — `stage_review.php` → `advance_sdl_next` с обновлением `current_stage` / `current_task` и опциональным `ClientNotifications::insert`.

**Глобальный поиск:** при `funds_status = global_searching` — клиенту показывается сценарий AML/реестров, не обещать мгновенный результат.

Подробный UI-гайд: `gsf_sdl_stage_guide.html`, техпособие `docs/01-technical/GSF_SDL_CLIENT_CABINET_IMPLEMENTATION_MANUAL_RU.md`.

## C. Документы

- **Справочник для оператора:** `/admin/document_guide.php` — слоты identity, legal, SOF по валютам (KZT/EUR/USD/crypto): что запросить, типичная путаница клиента.
- **Этап 1:** паспорт, селфи/liveness, ИИН/адрес — статусы `pending_review` / approved / rejected с понятной причиной отказа.
- **Этап 2:** зоны SOF на `source_of_funds.php`; пакет по сумме заявки (от справки до расширенного пакета на крупные суммы).
- **Этап 3+:** юридический пакет в `documents.php`.
- Клиенту при отказе: **что исправить**, без внутренних кодов `doc_key` в тексте уведомления (можно в ссылке на слот).

## D. Психология и коммуникация

**Портрет:** 30–55 лет, потеря от $1k, страх, стыд, недоверие, надежда на возврат.

**Золотое правило:** сначала признать боль и ситуацию — потом этапы и оплата.

**Доверие:** конкретные сроки этапов, прозрачность, кейсы; **не** обещать 100% возврат и точную дату выплаты без факта в БД.

**Язык:** зеркалить клиента («мошенники», «кидалы», «брокеры»); без CRM-жаргона в чате с клиентом.

**Оплата:** привязка к этапу SDL; сравнение комиссии с долей потери; срочность без агрессии.

**План влияния (AI):** из `psychotype` + stage + блокеры → стратегия → шаги (документы, `staff_tasks`, `client_notifications`) → целевые `funds_status` и `progress_percent`.

Источник углубления: `rabota/dla_posobia/ПОЛНОЕ_ПРАКТИЧЕСКОЕ_ПОСОБИЕ*.md`.

## E. Уведомления и действия клиента

| Тип | Поведение |
|-----|-----------|
| Обычное | Прочитать / ознакомиться |
| `ACTION_SIGN` | Клиент **подписывает биометрией** (`sign.php`, удержание ~2.5 с) — не «ознакомился» |
| `ACTION_BANK_LOAN` | Переход на `bank.php?loan=ID` — статус, документы, депозит |
| Крипто-вывод | `withdraw_crypto.php` — только USDT, **вся сумма** кувшина crypto после консолидации админом |

Тексты уведомлений: нейтральные статусы («На проверке», «Требуется подпись», «Откройте заявку»).

---

## F. Крипто-вывод (USDT)

1. Админ при необходимости консолидирует балансы в **USDT** (кувшин `crypto_balance`).
2. Публикует заявку клиенту → уведомление с действием подписи.
3. Клиент на `withdraw_crypto.php` видит сумму и подписывает.
4. Клиенту **не** показывать внутренний TX hash и операции `jar_adjust` админа.

---

## G. Банк / кредит под залог SDL

- Заявка от 1M ₸, LTV до 75%, залог — балансы SDL.
- Статусы: `pending_review`, `docs_required`, одобрение только админом.
- Клиент видит карточки на `bank.php`, загрузка документов по слотам из заявки.
- AI советует: какие документы добрать, напомнить про депозит/платёж, не обещать одобрение до статуса в БД.

---

## H. Подписываемые документы

- Шаблоны: contract, claim, report, custom — админ `sign_documents.php`.
- Клиент: `my_signatures.php` → биометрия.
- AI может рекомендовать `create_signable_document` только с осмысленным текстом для клиента (плейсхолдеры `{{fio}}`).

## I. Что советовать оператору по ситуациям

| Ситуация | Действие |
|----------|----------|
| Клиент на этапе 1, документы не загружены | Мягко объяснить зачем ID; ссылка на `documents.php`; не переходить на 2 |
| Документы на проверке | «Специалист проверяет, обычно 24–48 ч»; staff_task если >48 ч |
| SOF неполный | Указать конкретные зоны из `source_of_funds.php`; сверка с document_guide |
| Застрял на этапе 2 после принятия SOF | Проверить `sofAwaitingStage3Open`; вручную 2→3 + уведомление |
| Регуляторы / global_searching | Спокойный тон, этап идёт по регламенту, без дат «завтра выплата» |
| Несколько валют на счетах | Сначала консолидация кувшинов, потом вывод |
| Крипто | Только USDT, полный баланс, подпись клиента |
| Кредит docs_required | Список слотов + `bank.php?loan=` |
| Высокий risk_score | Усиленный комплаенс, не ускорять этапы без документов |
| Клиент нервничает | Признание эмоций + один следующий шаг, не весь пайплайн сразу |
