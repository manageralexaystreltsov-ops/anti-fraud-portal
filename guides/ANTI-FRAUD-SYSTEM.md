## 1. Зачем нужна Anti-Fraud система

Клиент, попавший в инвестиционное мошенничество, **не приходит с запросом «купить юридическую услугу»**. Он приходит с:

- страхом потерять ещё больше;
- убеждённостью, что «на экране лежат 50 000–500 000 $»;
- готовностью платить «последний взнос» мошенникам;
- стыдом («я не должен был вестись»);
- иногда — вторым кругом обмана (recovery scam).

**Anti-Fraud система** — это не «база про мошенников для Wikipedia». Это **операционная машина**:

| Этап | Что делает система | Результат |
|------|-------------------|-----------|
| **Учим** | Объясняем клиенту (и менеджеру), на какой стадии схемы он находится | Клиент перестаёт платить мошенникам |
| **Понимаем** | Снимаем психопрофиль, стадию, триггеры, сопротивление | Менеджер знает, *как* говорить, не *что* продавать вслепую |
| **Запускаем** | Ведём легальный процесс: документы, регулятор, банк, оплата услуг | Деньги идут **нам за работу**, а не мошенникам |

**Главный принцип:** мы смотрим на схему **глазами жертвы и исследователя**, не глазами мошенника. В CRM нет «скриптов обмана» — есть **распознавание стадий** и **легальный выход**.

## 2. Три задачи системы: учим → понимаем → запускаем

```
┌─────────────────────────────────────────────────────────────────┐
│                    ANTI-FRAUD SYSTEM                             │
├──────────────┬──────────────────────┬───────────────────────────┤
│   УЧИМ       │    ПОНИМАЕМ          │    ЗАПУСКАЕМ              │
│              │                      │                           │
│ Карта        │ Психопрофиль         │ CRM-цикл                  │
│ инстанций    │ Insights + Coach     │ Playbook                  │
│ Пособие      │ Стадия схемы (А–К)   │ Decision Engine           │
│ для клиента  │ Тип клиента          │ Jarvis + роли             │
│ Диагностика  │ Триггер / resistance │ Документы + PDF           │
│ по фразам    │ Гипотеза неоплаты    │ Заказы + хронология       │
└──────────────┴──────────────────────┴───────────────────────────┘
         │                  │                      │
         └──────────────────┴──────────────────────┘
                            │
                   Клиент не платит
                   мошенникам дальше
                   и идёт в легальный
                   процесс с нами
```

### 2.1. Учим — что входит

- **Публичное пособие** (`anti-fraud-kazakhstan.html`) — для клиента и родственников
- **Карта инстанций** — 10-секундная диагностика: «где вы сейчас»
- **Объяснение психологии** — почему «остался последний платёж» не значит «остался последний платёж»
- **Разделение** настоящих органов (АРРФР, банк, полиция) и фейковых «регуляторов» в WhatsApp

### 2.2. Понимаем — что входит

- Поля анкеты: брокер, потери ₸, стадия дела, отношение к оплате
- **client_psychology** — тип, триггер, сопротивление
- **Playbook** — Tight/Loose × Passive/Aggressive
- Jarvis Insights — «что клиент хочет на самом деле» (безопасность / контроль / надежда / доказательства)
- Привязка к **стадии мошеннической схемы** (новое поле — см. Roadmap)

### 2.3. Запускаем — что входит

- Этапы CRM-цикла: лид → консультация → документы → оплата → кейс → закрытие
- **Playbook step** → **Decision** → **Task** → **Document order**
- Jarvis генерирует тексты (Добряк + роли)
- PDF → менеджер отправляет клиенту вручную
- Хронология фиксирует каждый шаг для супервайзера

## 3. Архитектура: четыре слоя

Согласовано с архитектурой CRM Lichka (см. `docs/DAYLOG-2026-06-01.md`):

| Слой | Компоненты | Anti-Fraud функция |
|------|-----------|-------------------|
| **1. Память** | PostgreSQL, case_timeline, audit_log, golden fields | Факты: сколько потерял, кому платил, какая стадия |
| **2. Понимание** | Insights, Psychology, Coach Client/Manager | «Он на стадии налога, боится жены, tight_passive» |
| **3. Решения** | Router, Agents, Decision Engine, Playbook Engine | «Сначала остановить платежи мошенникам, потом KYC-документ» |
| **4. Действие** | Jarvis тексты, PDF, tasks, command center | Менеджер делает следующий шаг сегодня |

**Правило Anti-Fraud:** слой «Понимание» **обязан** знать стадию схемы до того, как слой «Решения» предложит документ или оплату.

## 4. Модуль 1 — Обучение и распознавание

### 4.1. Дерево мошеннических инстанций

Базовая цепочка (не все шаги у всех групп, но типовой порядок):

```
Платформа → Регулятор → AML → Налог → Юрист → Страховой фонд
    → Суд → Исполнитель → Центробанк → Recovery scam
```

**Ключевое правило для обучения:**

> Каждая новая инстанция появляется **только после оплаты предыдущей**.  
> AML пройден → нужен налог. Налог оплачен → нужен страховой депозит. И так далее.

### 4.2. Диагностика по фразам клиента

| Что говорит клиент | Стадия схемы | Что делать менеджеру |
|-------------------|--------------|---------------------|
| «Нужна верификация» | **А** — начало | Объяснить: платёж до вывода = мошенничество |
| «Регулятор проверяет счёт» | **Б** | Спросить: откуда письмо? WhatsApp или finreg.kz? |
| «Нужно пройти AML» | **В** | Показать карту: после AML всегда новый сбор |
| «Надо оплатить налог» | **Г** | Объяснить: налог платится после получения, не до |
| «Со мной работает юрист» | **Д** или проводник | Уточнить: он просил деньги сразу или «нашёл деньги»? |
| «Есть решение суда» | **Ж** | PDF с печатями ≠ настоящий суд; проверить через advokat.kz / e.gov |
| «Остался последний платёж» | **Финальная выкачка** | **Стоп-платежи.** Это не конец, это сигнал готовности платить |
| «Обещают вернуть деньги» | **К — Recovery** | **Стоп.** Второй круг обмана |

### 4.3. Что показываем клиенту (пособие)

Файл: `/home/vladymyr/Загрузки/anti-fraud-kazakhstan.html`

| Раздел | Содержание |
|--------|-----------|
| 1 | 7 стадий обмана (от знакомства до исчезновения) |
| 2 | **Карта движения по фейковым инстанциям** (18 шагов + А–К) |
| 3 | 12 красных флагов |
| 4 | Как проверить компанию (finreg.kz, whois) |
| 5 | Что делать, если уже попали |
| 6 | Реальные контакты (1434, полиция, e-gov) |

**Когда отправлять клиенту:** после первой консультации, когда он всё ещё верит «регулятору на экране». Формат: ссылка или PDF-экспорт.

### 4.4. Что объясняем про психологию (коротко)

Клиент продолжает платить **не потому что глупый**, а потому что:

1. **Уверен, что деньги на экране существуют** — 50 000 $ «уже его»
2. **Невозвратные затраты** — «я уже заплатил 2 000, ещё 500 и всё»
3. **Логика выгоды** — «заплачу 3 500, получу 120 000» (не «потеряю 3 500»)
4. **Документ с его фамилией** — PDF с суммой и номером дела = «доказательство»

Менеджер **не высмеивает** клиента. Добряк говорит: «Я понимаю, почему это выглядело логично».

## 5. Модуль 2 — Понимание клиента

### 5.1. Анкета — обязательные поля Anti-Fraud

| Группа | Поле | Зачем |
|--------|------|-------|
| **lichka** | `investment_loss_kzt` | Масштаб дела |
| **lichka** | `recovery_target_kzt` | Цель возврата (Lichka-поток) |
| **legal** | История с брокером / платформой | Название мошеннической платформы |
| **bank** | Источник средств, карта | Чарджбэк vs регулятор |
| **regulator** | Статус у регулятора | Что клиент *думает* про «регулятора» |
| **payment** | `payment_objection` | Почему не платит нам |
| **payment** | `promised_pay_date` | Обещанная дата |

**Новое поле (рекомендация Roadmap):** `scam_stage` — enum: `A,B,C,D,E,F,G,H,I,J,K,recovery,unknown`

### 5.2. Психопрофиль (client_psychology)

Четыре типа (покер-модель):

| Тип | Поведение | Как вести |
|-----|-----------|-----------|
| **tight_passive** | Осторожный, боится ошибиться | Безопасность → кейсы → документы → оплата |
| **tight_aggressive** | Давит, проверяет каждое слово | Контроль → процесс → доказательства → решение |
| **loose_passive** | Эмоциональный, доверчивый | Контакт → эмоции → поддержка → план |
| **loose_aggressive** | Импульсивный, «давайте быстрее» | Срочность → выгода → дедлайн → оплата |

**Anti-Fraud нюанс:** клиент на стадии **«остался последний платёж»** часто **loose_aggressive** — нужен жёсткий стоп-платежи + контроль процесса.

### 5.3. Вопросы для первичной диагностики (скрипт менеджера)

Задавать **до** предложения документов:

1. **Платформа:** как называется? как давно? сколько вложили?
2. **Вывод:** пробовали вывести? что написали?
3. **Комиссии:** сколько раз просили доплатить? за что (AML, налог, страховка)?
4. **Документы:** присылали PDF? от кого (WhatsApp, email)?
5. **Юрист:** звонил кто-то «помочь»? просил деньги сразу?
6. **Сейчас:** что говорят «следующий шаг»? есть ли «последний платёж»?
7. **Родные:** знают ли? (изоляция — красный флаг мошенников)

Ответы → Jarvis в чат → автозаполнение анкеты → стадия `scam_stage`.

### 5.4. Coach — что система «понимает»

Coach Client анализирует:

- на каком **playbook step** клиент застрял;
- **resistance** — сопротивление оплате *нам* (не мошенникам);
- **trigger** — что подтолкнёт к действию (страх, надежда, жена, суд).

Coach Manager подсказывает менеджеру:

- не предлагать «налог 13%» как мошенники — предложить **легальный** пакет документов;
- не давить на tight_passive;
- на recovery scam — **не брать предоплату «за возврат»** без договора и без проверки.

## 6. Модуль 3 — Запуск процесса

### 6.1. CRM-цикл (7 этапов)

| # | Этап | Anti-Fraud действие |
|---|------|---------------------|
| 1 | **Новый лид** | Диагностика стадии А–К, стоп-платежи мошенникам |
| 2 | **Консультация** | Пособие, объяснение карты инстанций |
| 3 | **Сбор документов** | KYC: паспорт, выписки, скрины переписки с мошенниками |
| 4 | **Договор** | Договор Legal Consult / HRL |
| 5 | **Оплата услуг** | Заказ в `client_document_orders` — **наши** услуги |
| 6 | **Ведение кейса** | Жалобы АРРФР, банк, полиция — по ролям Jarvis |
| 7 | **Закрытие** | Архив или win-back |

### 6.2. Алгоритм «запуск процесса» (10 шагов)

```
1. Создать клиента в CRM (ФИО + телефон)
2. Вставить переписку в Jarvis → заполнить анкету
3. Определить scam_stage (А–К) по диагностике
4. Заполнить psych_profile → автовыбор Playbook
5. Decision Engine → «следующее действие»
6. Добряк: текст «остановите платежи мошенникам»
7. Создать заказ на первый документ (KYC / договор / жалоба)
8. После оплаты → PDF → отправить клиенту
9. Хронология: каждый контакт, каждый документ
10. Супервайзер: drift playbook, conversion
```

### 6.3. Стоп-правила перед любым «запуском»

| # | Правило | Почему |
|---|---------|--------|
| 1 | **Не создавать заказ** без явного приказа менеджера | Jarvis не выдумывает оплату |
| 2 | **Не обещать возврат** суммы с экрана мошенников | Эти деньги не существуют |
| 3 | **Не имитировать** фейковый «регулятор» для давления | Мы — легальная компания |
| 4 | **Recovery scam** — отдельный протокол, см. §10 | Второй круг обмана |
| 5 | **Золотые поля** — только SuperAdmin или approve proposal | Audit |

## 8. Сценарии по стадиям А–К

### Стадия А — «Не могу вывести»

**Сигналы:** ошибка системы, верификация, 100–1 000 $ или 5–10%.

**Учим клиента:**
- Настоящие платформы не берут деньги **до** вывода
- Сумма на экране — **нарисованная**

**Запускаем:**
- Jarvis (Добряк): мягко объяснить ловушку
- Зафиксировать `investment_loss_kzt`
- Playbook step 1: **безопасность**
- Не создавать заказ, пока клиент не понял стоп-платежи

**Decision Engine:** `STOP_SCAM_PAYMENTS` → задача менеджеру «объяснить карту инстанций»

---

### Стадия Б — «Регулятор проверяет»

**Сигналы:** письмо, сайт, ЛК с именем и суммой.

**Учим:**
- АРРФР не пишет в WhatsApp
- Проверка: finreg.kz, ardfm.gov.kz — только официальные домены
- «Независимый регулятор» платформы = актёр мошенников

**Запускаем:**
- Роль Jarvis **ardfm** — черновик *настоящей* жалобы (не имитация фейка)
- Заказ: «Жалоба в АРРФР» / KYC-пакет
- Хронология: приложить скрины фейкового «регулятора»

---

### Стадия В — AML

**Сигналы:** «международная проверка», 5–20%.

**Учим:**
- AML у настоящих органов **не оплачивается** переводом на карту физлица
- После AML **всегда** появится новая причина

**Запускаем:**
- Роль **afm** — объяснить разницу (для менеджера, не копировать мошенникам)
- Собрать выписки для **легального** AML-расследования
- Playbook: **доказательства** (tight_aggressive) или **поддержка** (loose_passive)

---

### Стадия Г — Налог

**Сигналы:** «13%», «18%», «20%», «осталось оплатить налог».

**Учим:**
- Налог на доход — **после** получения денег
- «Налоговый акт» в PDF от неизвестного email = подделка

**Запускаем:**
- Документ: справка / расчёт **легальной** стоимости услуг
- Добряк: «То, что просили у вас — не налог, а комиссия мошенников»

---

### Стадия Д — «Со мной юрист»

**Сигналы:** «единственный нормальный человек», звонок, сайт юрфирмы.

**Учим:**
- Настоящий юрист не «находит деньги на платформе-мошеннике» за 2 дня
- Юрист-мошенник **не просит** сразу — сначала надежда, потом проводник по цепочке

**Запускаем:**
- **Мы** — легальный юридический процесс (Добряк + договор)
- Не конкурировать с фейковым юристом «кто лучше найдёт деньги»
- Конкурировать: «мы остановим кровотечение и запустим реальное дело»

---

### Стадии Е–И — Страховка, суд, исполнитель, ЦБ

**Общий паттерн:** каждый шаг = «ещё чуть-чуть».

**Учим:** показать **всю цепочку** — клиент видит, что он на шаге 9 из 15, не на «последнем».

**Запускаем:**
- Роль **court** — только для подготовки **настоящих** документов в суд/АРРФР
- PDF с нашими шаблонами (dla_posobia), не подделки мошенников
- Command Center: приоритет «стоп-платежи» > «новый документ»

---

### Стадия К — Recovery scam

См. отдельный §10.

## 9. Роль «юриста» в схеме мошенников vs наша роль Добряка

| | Фейковый «юрист» | Добряк (Legal Consult) |
|---|-----------------|------------------------|
| **Когда появляется** | После блокировки вывода | Когда клиент ищет помощь |
| **Первый контакт** | «Нашёл ваши деньги» | «Расскажите, что произошло» |
| **Деньги сразу** | Обычно нет (надежда) | Прозрачный договор + заказ в CRM |
| **Функция** | Проводник по 10–15 «инстанциям» | Остановить обман + легальный процесс |
| **Документы** | PDF с вашим именем (фейк) | PDF из CRM, audit, golden fields |
| **Конец** | Исчезает с остальными | Хронология, закрытие дела |

**Фраза для обучения клиента:**

> Мошенник-юрист продаёт **надежду**. Мы продаём **процесс** — с договором, сроками и реальными органами.

## 10. Recovery Scam — отдельный протокол

### 10.1. Признаки

- Клиент **уже понял**, что первую платформу обманули
- Звонят «международные адвокаты», «детективы», «crypto recovery»
- «Нашли ваши деньги на транзитном счёте / в блокчейне»
- Просят **предоплату** за возврат

### 10.2. Протокол CRM

| Шаг | Действие |
|-----|----------|
| 1 | Пометить `scam_stage = recovery` |
| 2 | **Запрет** на любые заказы «за возврат потерянного» без договора HRL |
| 3 | Добряк: «Это второй круг. Не платите никому» |
| 4 | Отправить пособие § Recovery |
| 5 | Если клиент уже заплатил recovery-мошенникам — добавить в `investment_loss_kzt` |
| 6 | Полиция + e-gov — приоритетнее новых «юристов» |

### 10.3. Красные флаги recovery

- Юрист **сам** нашёл клиента
- Гарантия «100% возврат за 30 дней»
- Оплата только криптой / на карту физлица
- Нет юрадреса в РК или проверяемого БИН

## 15. Регулятор-симулятор

**URL:** `/regulator` или вкладка клиента Alt+5

**Назначение:** тренировка менеджера — **холодные отказы** АРРФР, KYC, KZT.

**Anti-Fraud использование:**

- Отработать возражения **до** разговора с клиентом
- **Не** отправлять клиенту текст симулятора как «ответ регулятора»
- Различать: симулятор (обучение) vs finreg.kz (реальность)

**Не путать:**

| Ситуация | Куда |
|----------|------|
| Чарджbэк карты | **Банк** (роль bank) |
| Жалоба на брокера | **АРРФР** (роль ardfm) |
| Мошенничество | **Полиция**, ст. 190 УК РК |
| Фейковый «регулятор» платформы | **Никуда** — стоп-платежи |

## 19. Чеклисты

### 19.1. Перед первым заказом документа

- [ ] Клиент понимает: сумма на экране платформы **не реальна**
- [ ] Клиент **остановил** платежи мошенникам (или подтвердил намерение)
- [ ] Стадия схемы зафиксирована
- [ ] Психопрофиль / playbook выбран
- [ ] Нет активного recovery scam без протокола
- [ ] Золотые поля проверены (потери ₸, брокер)

### 19.2. 12 красных флагов (из пособия)

1. Гарантированная прибыль  
2. 3–20% в день  
3. Нет лицензии АРРФР (finreg.kz)  
4. **Платёж ДО вывода**  
5. Давление «только сегодня»  
6. Только мессенджеры  
7. «Не говорите родственникам»  
8. «Возьмите кредит»  
9. Нет реального адреса  
10. Фейковые отзывы  
11. **Юрист сам позвонил**  
12. Домен < 1 года  

### 19.3. После каждого контакта

- [ ] Заметка или Jarvis sync в хронологию  
- [ ] Playbook step обновлён  
- [ ] Command center: «контакт был»  
- [ ] Если клиент снова платит мошенникам — escalate
